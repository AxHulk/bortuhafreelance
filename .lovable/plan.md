# Переработка раздела «Портфолио»

## Концепция

Двухуровневая иерархия в духе архитектурных бюро (Norm Architects, Studio KO):

```text
/portfolio                  → витрина ОБЪЕКТОВ
/portfolio/:object          → страница объекта со списком помещений
/portfolio/:object/:room    → editorial-страница помещения с фото и описанием
```

Editorial-стиль: крупная типографика, много воздуха, фото-герой во всю ширину, текст и фото сменяют друг друга в ритме лонгрида. Поддерживает 1–10 фото без визуальных «дыр».

---

## Структура страниц

### 1. `/portfolio` — Витрина объектов

Сетка крупных карточек объектов (а не помещений):

```text
┌──────────────────┐  ┌──────────────────┐
│  Большое фото     │  │  Большое фото    │
│  обложки объекта  │  │  обложки объекта │
├──────────────────┤  ├──────────────────┤
│ ЖК Пятый элемент  │  │ Гостевой домик   │
│ 75,8 м² · 2026    │  │ с сауной         │
│ 5 помещений →     │  │ 84,2 м² · 2025   │
└──────────────────┘  └──────────────────┘
```

Над сеткой — фильтры по типу (Жилые / Коммерческие / Курортные / 3D-визуализация) и по статусу (Реализован / Концепт / В процессе).

### 2. `/portfolio/:object` — Страница объекта

- Hero-баннер: панорамное фото, название объекта, ключевые мета (площадь, локация, год сдачи, статус).
- Короткое введение об объекте (2–4 строки).
- Сетка карточек помещений: квадратное фото + название помещения + площадь. Клик → страница помещения.
- Если у объекта пока 1 помещение — показывает одну карточку, и всё работает корректно.

### 3. `/portfolio/:object/:room` — Editorial-страница помещения

Структура лонгрида (показывается ровно столько, сколько есть фото — от 1 до 10):

1. **Хлебные крошки**: Главная → Портфолио → Объект → Помещение
2. **Hero-фото** во всю ширину (1-е фото галереи) с overlay-заголовком: «Детская» + площадь.
3. **Мета-блок**: Объект · Площадь · Стиль · Статус · Год.
4. **Введение / описание** — крупный текст в узкой колонке (max-width ~640px), как в журнале.
5. **Editorial-галерея** — фото чередуются по паттерну, ритм меняется в зависимости от количества:
   - 1 фото — крупное во всю ширину (уже в hero).
   - 2–3 фото — full-width, с подписью под каждым.
   - 4–6 фото — чередование full-width и 2-колоночных блоков.
   - 7–10 фото — добавляются «галерейные строки» по 2 фото в ряд + 1 акцентное full-width.
6. **Подпись/задача** в спокойной плашке между фото.
7. **Отзыв клиента** (если есть) — типографика крупным курсивом, без рамок.
8. **Навигация по помещениям объекта**: «← Предыдущее помещение | Все помещения объекта | Следующее →».
9. **CTA**: «Хотите похожий проект?» → опросник.

## Модель данных

Текущий тип `Project` соответствует помещению. Вводим новый верхний уровень `PortfolioObject` и переименовываем `Project → Room`. Новый файл `src/data/objects.ts`:

```ts
export interface PortfolioObject {
  id: string;                       // "zhk-pyatiy-element"
  title: string;                    // "ЖК Пятый элемент"
  area: string;                     // "75,8 м²"
  location: string;
  year: string;                     // "Сдача апрель 2026"
  status: "Реализован" | "Концепт" | "В процессе";
  category: ProjectCategory;
  intro: string;                    // 2–4 строки об объекте
  coverImage: string;
  roomIds: string[];                // ссылки на Room.id
}
```

Каждой существующей записи (бывший Project, теперь Room) добавляется поле `objectId: string`. Я не угадываю принадлежность — оставляю значения пустыми/single-room обёрткой, **вы укажете руками** (как договорились). Стартовый набор объектов:

- `zhk-pyatiy-element` — пока пустой контейнер, заполните по мере добавления помещений.
- `guest-house-sauna` — пока пустой контейнер.
- Для всех текущих 12 записей создаются «обёртки-объекты» 1:1 (`object_<id>`), чтобы ничего не сломалось — позже вы сольёте их вручную, переназначив `objectId` у нужных Room.

Обратная совместимость URL: старые `/portfolio/:roomId` редиректят на `/portfolio/:objectId/:roomId`.

**Производительность:**
- Использовать существующие WebP-thumbs для обложек и сетки.

**Ничего не ломаем:** старые ссылки `/portfolio/:id` работают через редирект.

## Что я НЕ делаю в этой итерации

- Не группирую существующие записи по объектам автоматически — вы пришлёте принадлежность списком после утверждения плана, либо отдельным сообщением. До этого каждая запись = свой объект-обёртка (структура и UI уже работают корректно).
- Не загружаю детскую из docx и фото из `0006.png` — это отдельная задача «добавить помещение в объект ЖК Пятый элемент», сделаю следующим шагом по вашему запросу.
